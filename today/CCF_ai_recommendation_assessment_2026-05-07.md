# CCF — 자연어 + AI 추천(ICD/검사/오더/처방) 제품 평가

> 작성일: 2026-05-07
> 맥락: `docs/CCF_strategic_decision_2026-05-07.md` 작성 후, 추가 제품 아이디어 검토
> 결론 요약: **"검색형(retrieval-based) + 의사 확인"** 아키텍처에서는 제품 위험도가 크게 낮아지고, 단위 경제·가치 명제 모두 클릭 차트보다 우월할 수 있다.

## 0. 평가 대상 제품

```
의사의 자유 텍스트 입력
        ↓
[추출 + 검색 + 랭킹 엔진]
        ↓
상위 N개 후보 제시:
  - ICD 코드 (KCD-8 기준)
  - 검사 (영상/검사실)
  - 오더 (시술/처치)
  - 처방 (약물 + 용량)
        ↓
의사가 클릭으로 확정 (또는 다른 코드/처방 직접 선택)
        ↓
의사의 과거 패턴으로 랭킹 개인화
```

이 문서는 **이 제품의 실현 가능성·ROI·클릭 차트와의 관계**를 검토한다.

---

## 1. 핵심 프레이밍 — 생성형이 아닌 검색형

이 제품의 위험도와 비용은 **아키텍처 선택에 결정적으로 좌우**된다.

| 차원 | 생성형 (Generative) AI 추천 | **검색형 (Retrieval) AI 추천** |
|---|---|---|
| 추천 풀 | LLM이 자유롭게 추론 | EMR DB의 유효 항목만 |
| 환각 리스크 | 존재하지 않는 약/코드 생성 가능 | **불가능** (DB 풀 외 추천 못 함) |
| 응답 속도 | 1~3초 (LLM 추론) | **밀리초** (벡터 검색) |
| 호출당 비용 | $0.05~0.30 | **$0.001 미만** |
| 규제 분류 | SaMD 가능성 | "검색·자동완성 도구"로 회피 가능 |
| 책임 소재 | "AI가 결정" 분쟁 가능 | "의사가 검색 결과 중 선택" 명확 |

## 2. 검색형 아키텍처의 구조

```
[입력] 자유 텍스트 / 음성
    ↓
[Step 1] 자유 텍스트 → 구조화 쿼리
    - LLM 또는 NER 모델 (예: "허리통증, 다리저림" → 키워드 추출)
    - 이 단계가 시스템 정확도 천장
    ↓
[Step 2] 후보 검색 (Retrieval)
    - KCD-8 코드 DB, KIMS 약 DB, 심평원 청구 코드 DB에서 검색
    - 임베딩 기반 시맨틱 검색 또는 키워드 검색
    ↓
[Step 3] 랭킹 (Ranking)
    - 의사 본인의 과거 사용 빈도 (개인화)
    - 일반 빈도 (콜드 스타트 시)
    - 환자 컨텍스트 적합성 (나이, 성별, 동반질환)
    ↓
[Step 4] 상위 N개 표시
    - top-5 또는 top-10
    - 각 항목에 메타정보 (사용 빈도, 청구 거절률 등)
    ↓
[Step 5] 의사 선택 또는 자유 입력
    - 후보가 맘에 안 들면 직접 검색·입력
    - 의사 결정 = 시스템 학습 데이터 (다음 랭킹에 반영)
```

**핵심**: AI는 **"검색하고 정렬"**만 한다. 결정은 의사가 한다.

## 3. 이전 평가에서 과대평가했던 위험들

### 3-1. "할루시네이션 → 환자 안전" → 대부분 해소
- 검색형은 존재하지 않는 약/코드를 만들어낼 수 없음 (DB에 있는 것만 추천)
- 잘못된 용량 → KIMS DB가 답을 가지고 있으니 시스템이 잘못 표기 안 됨
- 의사가 마지막 결정 → AI는 결정 안 함
- **남는 위험**: 적절하지 않은 코드를 추천 → 의사가 무비판 수용 (이건 자동완성 일반의 문제)

### 3-2. "의료기기(SaMD) 분류" → 회피 가능성 높음
- "검색·제안 도구" 포지셔닝으로 비-의료기기 가능성 큼
- 기존 EMR의 자동완성과 본질적으로 같은 카테고리
- **남는 위험**: 식약처 해석 변경 가능성 → 사전 자문 1회 필요

### 3-3. "LLM 비용·지연" → 현실적 통제 가능
- 핵심 추천은 검색(벡터/키워드) → 호출당 0.001원 미만
- LLM은 **자유 텍스트 → 쿼리 변환**에만 사용 (또는 아예 안 써도 됨)
- 의사당 변동비 추정: **연 30~80만원** (이전 추정 155~600만원에서 대폭 하향)

### 3-4. "한국 의료 특수성" → 사실은 강점
- KCD-8, 심평원 코드, KIMS DB는 공개 또는 구매 가능
- 검색 풀 자체가 정답을 가지고 있으니 정확도 100%
- 글로벌 LLM이 한국 코드 외운 것보다 **DB 검색이 압도적으로 정확**
- → 한국 시장 해자

### 3-5. "훈련 데이터 부족" → 별 문제 안 됨
- 검색형은 레이블된 학습 데이터 거의 불필요
- 개인화 시그널 = 의사 본인의 과거 진단·처방 빈도 (단순 통계)
- 콜드 스타트는 일반 빈도로 시작, 사용할수록 개인화

---

## 4. 그럼에도 남는 진짜 위험

### 4-1. 자유 텍스트 → 쿼리 변환 정확도 ★ 새 핵심
- "허리가 아프고 다리가 저린 환자" → 어떤 ICD 후보를 검색할지 결정하는 단계
- 단순 키워드 매칭으론 부족
- LLM 또는 의료 NER 모델 정확도가 시스템 천장
- **완화**: 잘못 검색되어도 의사가 무시 가능 → 안전 마진 있음

### 4-2. 랭킹 품질의 함정
- 후보 50개 중 정답이 47번째에 있으면 무용지물
- top-5 안에 정답을 넣어야 가치 있음
- 의사 패턴이 적은 초기엔 일반 빈도로 시작 → 점진 개선

### 4-3. 청구 코드 거절·삭감 책임
- 의사가 잘못된 코드를 클릭해도 의사 책임
- 하지만 "AI가 추천했다"는 분쟁 시나리오 가능
- **완화**: 추천 코드와 함께 청구 거절률·삭감률 메타정보 노출

### 4-4. EMR DB 접근권 (운영 문제)
- KCD-8, KIMS, 심평원 코드는 공개·구매 가능
- 의사 본인의 과거 차트(개인화 시그널)는 EMR에 묶임
- → 클릭 차트와 같은 데이터 export 문제 존재

### 4-5. 경쟁
- 기존 EMR 자동완성과 차별화는 **개인화 깊이 + EMR 횡단 + UX 완성도**
- 네이버·카카오 진입 시 자본 차이 큼
- 차별화 영역: **정형외과 버티컬 + 한국 청구 코드 + 의사별 깊은 개인화**

## 6. 클릭 차트와의 통합 — SOAP 영역별 UX 분담

가장 중요한 인사이트: 두 제품은 경쟁재가 아니라 **SOAP 영역별로 자연스럽게 분담**된다.

| SOAP 영역 | 적합한 UX | 이유 |
|---|---|---|
| **S (Subjective)** — 증상, 호소 | 클릭 차트 | 관찰 기반, 칩 클릭 자연스러움 |
| **O (Objective)** — 검진, 검사 결과 | 클릭 차트 + **검사 추천** | 부위별 검진은 클릭, 검사 오더는 추천 |
| **A (Assessment)** — 진단 | **ICD 검색·추천** | 가치 가장 큼 (청구 정확도 직결) |
| **P (Plan)** — 처방, 오더 | **처방·오더 검색·추천** | 두 번째로 가치 큼 |

**통합 UX 흐름 예시:**

```
[S] 클릭으로 호소 입력
    "허리, 우측 엉치, 다리 당김, 1주 전 발병"
        ↓
[O] 클릭으로 검진 결과
    "SLR(45/full), Motor 정상, Sensory L5 dec"
        ↓
[A] 시스템이 ICD 후보 추천 (의사 컨텍스트 + 본인 빈도 기반)
    1. M51.1 추간판 탈출증 (요추) — 본인 사용 빈도 1위
    2. M54.5 요통 — 일반 빈도 상위
    3. M48.06 척추협착증 (요추) — 컨텍스트 적합
    ...
    → 의사 클릭으로 선택 (또는 자유 입력)
        ↓
[P] 선택된 진단에 따른 처방·오더 추천
    1. 표준 패키지: PT + PO 1wk + FU 1wk (본인 빈도)
    2. 변형: + SNRB L4-5
    3. ...
    → 의사 가감
```

→ 자유 텍스트 입력 부담 없음, 동시에 추천 자동화. **단일 제품 경험**.

## 7. 시나리오 비교 (이전 문서 보강)

이전 문서의 시나리오 A/B/C에 추가하여:

### 시나리오 Y' — 클릭 차트 + 검색형 추천 동시 통합 출시
- S/O는 클릭, A/P는 검색 추천
- 단일 통합 제품
- **출시 기간 6~12개월** (클릭 차트 단독 6주 → 통합 6~12개월)
- **가치 명제 압도적 강함** ("입력 빠름 + 청구 정확")
- 차별화 명확 (개인화된 청구 코드 추천)

### 시나리오 Z' — 검색형 추천을 메인, 클릭 차트는 보조
- 자유 텍스트 입력 → 시스템 추출 → 클릭 확정
- 사실상 **"AI 보조 차팅 + 보험 청구 자동화"**
- ambient AI(Nuance DAX 류)에 가까운 UX이지만 검색형으로 안전성 확보
- 가치 가장 강함, 개발 난이도 가장 높음

### 시나리오 비교표

| 시나리오 | 출시 기간 | 가치 강도 | 위험도 | 자본 부담 |
|---|---|---|---|---|
| A (클릭만) | 6~8주 | ★★ | ★ | 낮음 |
| B (마이닝→클릭) | 6개월+ | ★★★ | ★★★ | 중 |
| C (마이닝 수평 역량) | 12개월+ | ★★★★ | ★★★ | 중상 |
| **Y' (클릭+검색 통합)** | **6~12개월** | **★★★★** | **★★** | **중** |
| Z' (검색 메인) | 12~18개월 | ★★★★★ | ★★★ | 중상 |

## 8. 분기 질문 — 의사 인터뷰로 확인

### 핵심 질문
> **"의사에게 가장 강한 가치 명제는 입력 속도(클릭 차트)인가, 청구 정확도(AI 추천)인가?"**

### 인터뷰 가설
- 가설 H1: 입력 속도가 핵심 → 시나리오 A 또는 Y' 합리적
- 가설 H2: 청구 정확도가 핵심 → 시나리오 Y' 또는 Z' 합리적
- 가설 H3: 둘 다 비슷하게 중요 → 시나리오 Y' 압도적

### 인터뷰 방법
- 의사 5~10명, 정형외과 개원의 우선
- 시나리오 모형(목업) 제시 후 선호도 측정
- 청구 거절·삭감 경험 빈도 직접 질문 (가치 정량화)
- 차팅 시간 vs 청구 정확도의 trade-off 시나리오 제시

## 9. 기술적 구현 가능성 평가

### 가능성 (검색형 아키텍처 기준)
| 컴포넌트 | 난이도 | 비고 |
|---|---|---|
| KCD-8/KIMS/심평원 DB 통합 | 중 | 공개·구매 가능, 형식 정규화 필요 |
| 임베딩 검색 (벡터 DB) | 낮 | 표준 컴포넌트 (Pinecone, Qdrant 등) |
| 자유 텍스트 → 쿼리 변환 | 중상 | LLM 또는 fine-tuned NER 필요 |
| 개인화 랭킹 | 중 | 빈도 통계 + LTR (Learning to Rank) |
| UI 통합 (클릭 차트 + 추천) | 중 | 디자인 도전 |
| 컴플라이언스 인프라 | 중상 | 클릭 차트 대비 큰 차이 없음 |

→ **기술적 구현은 충분히 가능**. 클릭 차트 단독보다 +6개월 정도 추가.

### 자본 추정
- 시나리오 Y' (통합 출시): 클릭 차트 대비 1.5~2배 (인력·기간)
- 핵심 비용: NLP 엔지니어 1~2명, 의료 데이터 라이선스, 임베딩 인프라

## 10. 예상 정확도 (검색형 기준)

| 단계 | 정확도 추정 |
|---|---|
| Step 1 자유 텍스트 → 쿼리 변환 | 80~90% (LLM) / 65~80% (룰 기반) |
| Step 2 후보 검색 (DB 적중) | 100% (DB가 답을 가지고 있음) |
| Step 3 랭킹 top-5 정답 포함 | **75~88%** (의사 데이터 풍부 시) |
| Step 3 랭킹 top-1 정답 일치 | 40~55% |
| Step 3 랭킹 top-5 (콜드 스타트) | 55~70% |

→ top-5 정확도는 자동완성 UX로 **충분히 사용 가능**한 수준.

## 11. 솔직한 강점 (재평가)

### 11-1. 가치 명제가 직접적
- "보험 청구 정확도"는 의사가 즉시 이해. 영업 메시지 강함
- 청구 거절·삭감 경험은 정형외과 의사 거의 모두가 가진 페인 포인트

### 11-2. 차별화 영역 명확
- 개인화 깊이 + 한국 보험 코드 + 정형외과 버티컬 = 글로벌 LLM도 못 따라옴
- 클릭 차트는 "더 빠른 입력"인데 비해, 이 제품은 "더 정확한 청구" → 가치 차원이 다름

### 11-3. 단위 경제 우월
- 검색형으로 변동비 통제 → 마진 +170~+720만원/년
- 클릭 차트(+60~+90)보다 우월

### 11-4. 의사 락인 강함
- 의사 본인 사용 패턴 = 개인화 시그널
- 시간 지날수록 추천 정확도 ↑ → 이탈 비용 ↑

### 11-5. 클리닉 단위 확장 자연
- 청구 정확도는 클리닉 운영진이 직접 가치 인지
- 원장 의사가 페이닥터 도입 추천 동기 강함

## 12. 솔직한 약점 (재평가)

### 12-1. 자유 텍스트 이해의 천장
- Step 1 정확도가 시스템 천장

- 처음에는 모든 추천 검증 → 시간 절감 효과 적음
- 신뢰 쌓이는 데 3~6개월
- 이 기간 동안 이탈 가능성

### 12-3. 한국 청구 규칙 변경 빈번
- 심평원 청구 규칙·약가·급여 기준 자주 바뀜
- DB 갱신 인프라 + 운영팀 필요
- → 운영 부담 (분기별 갱신 + 모니터링)

### 12-4. EMR 통합 운영 마찰
- 의사 본인 차트 데이터 확보는 클릭 차트와 같은 문제
- EMR 별 export 포맷 상이

### 12-5. 의사가 잘못된 코드 클릭 시 책임 분쟁
- 검색형이라도 분쟁 가능성 존재
- 추천 메타정보(거절률 등) 표시로 완화
- 보험 가입 등 운영적 대응 필요

## 16. 결론 한 줄

> **"검색형 + 의사 확인" 아키텍처는 생성형 AI의 위험을 대부분 해소**하며,
> **단위 경제도 클릭 차트보다 우월**할 가능성이 있고, **가치 명제(청구 정확도)는 압도적으로 강하다**.
> 따라서 **시나리오 Y' (클릭 차트 + 검색 추천 통합)**는 단순 클릭 차트(시나리오 A)보다 **ROI가 더 높을 수 있는 진지한 후보**다.
> 단, **의사 인터뷰 5~10명으로 가치 우선순위 확인**이 시나리오 결정의 전제다.
