# CCF — 의사별 마이닝 기반 온보딩 계획

> 최종 개정: 2026-05-07
> 입력 데이터: 의사별 과거 차팅 데이터 (예: `visit_timeline.csv` 12,284건은 한 의사 샘플)
> 스키마: `docs/CCF_SOAP_charting_requirements.md`
> 케이스 분해 포맷: `docs/CCF_SOAP_case_decomposition.md`
> 미래 강화책: `docs/CCF_charting_markov_design.md` (출시 후 클릭 로그 누적 시점)

---

## 0. 핵심 프레이밍

이 계획은 **범용 룰북**이 아니라 **의사 1인당 1 룰북**을 자동 생성하는 **온보딩 서비스**의 설계 문서다.

```
신규 의사 가입 → 그 의사의 과거 차트를 받음 →
자동 마이닝 → 의사 본인이 짧게 큐레이션 →
의사 전용 룰북 생성 → CCF UI에 로딩 → 사용 시작
```

이 프레이밍에서 **"단일 의사 데이터"는 한계가 아니라 제품 그 자체**다. 대신 **콜드 스타트, 데이터 수급, 개인정보 처리, 의사당 가변비** 같은 **새로운 종류의 한계**가 등장한다. 이 문서는 그것들을 정직하게 짚는다.

---

## 1. 제품 포지셔닝

### 가치 제안
> "당신이 써온 차트 표현 그대로, 첫날부터 최적화된 클릭 차팅."

일반 EMR 템플릿이 "남이 만든 양식"이라면, CCF는 "당신의 차팅 패턴을 그대로 옮긴 양식"이라는 차별화.

### 고객 단위
- **1 의사 = 1 고객 = 1 룰북** (개인 단위 소유)
- 클리닉 전체 도입 시도 의사별 룰북이 병렬 운영
- 이직 시 룰북은 의사 개인을 따라 이동 (계약 정리 필요 — §16)

### 경쟁 포지셔닝
- 기존 EMR 템플릿: 일반화된 양식, 개인화 거의 없음
- 음성 인식 차팅: 빠르지만 구조화 안 됨
- AI 자동 차팅: 정확도 미흡, 책임 소재 모호
- **CCF 의사별 마이닝**: "의사 본인 패턴 = 책임 소재 명확 + 친숙한 표현 + 빠른 입력"

## 2. 고객 여정 (의사 한 명의 온보딩 흐름)

```
[D-7]  의사 가입 → 계약 + PHI 처리 동의
[D-6]  과거 차트 export 안내 (가장 어려운 단계)
[D-3]  데이터 도착 → 비식별화 + 정규화 → 자동 마이닝 시작
[D-1]  마이닝 완료 → 의사에게 "30분 큐레이션" 안내
[D-0]  의사 본인 큐레이션 (UI 안에서 칩 추가/제거/순서 변경)
[D+0]  실사용 시작
[D+30] 첫 클릭 로그 분석 → 룰북 자동 갱신 알림
[D+90] 정기 갱신 + 사용성 메트릭 리뷰
```

가장 큰 마찰 지점:
- **D-6 데이터 export**: EMR 별 포맷 상이, 행정 절차 길어질 수 있음
- **D-0 큐레이션 UX**: 너무 길면 이탈, 너무 짧으면 품질 하락

## 3. 데이터 자산 점검 (의사별)

| 컬럼 | 활용 가치 | 의사별 마이닝 관점 |
|---|---|---|
| 환자번호, 환자명 | 낮음 | 비식별화 후 단순 case ID로 대체 |
| 내원일 | 중간 | 시기별 패턴 변화 추적용 |
| **증상·차팅** | **매우 높음** | 핵심 마이닝 대상 (S/O/A/P 추출) |
| **진단목록** | **매우 높음** | 자유 텍스트 파싱 없이 바로 사용 가능 — A 단계 후보군 |
| **처방목록** | **매우 높음** | 진단×처방 페어 마이닝의 직접 입력 |
| 진단수, 처방수 | 중간 | 케이스 복잡도 필터링 |

> **핵심 발견(이전 문서와 동일)**: `진단목록 × 처방목록` 페어는 자유 텍스트 파싱 없이 즉시 마이닝 가능. 의사별로도 가장 빠른 가시 성과를 내는 영역.

## 4. 마이닝 산출물 (의사별 3종)

이전 문서의 3종이 이제 **의사별 디렉토리**에 저장된다.

### (1) `rules/{doctor_id}/flow.json` — 부위별 표준 입력 흐름
의사가 평소 어떤 순서로 차팅하는지 추출.

### (2) `rules/{doctor_id}/chips.json` — 항목별 빈출 칩 후보
의사가 자주 쓰는 표현·약어·진단명을 빈도순으로.

### (3) `rules/{doctor_id}/diagnosis_packages.json` — 진단별 표준 처방 패키지
이 의사가 특정 진단에 보통 어떤 처방 묶음을 쓰는지.

각 파일에는 메타데이터 포함:
```json
{
  "doctor_id": "d_12345",
  "mined_at": "2026-05-07",
  "source_case_count": 12284,
  "source_period": "2023-12 ~ 2026-04",
  "version": "1.0",
  "curated_by": "d_12345 self",
  "curated_at": "2026-05-08"
}
```

## 5. 마이닝 파이프라인 (의사별 5단계)

```
[Phase 0] 의사 단위 토큰 사전 적용 (공통 vocabulary + 의사 약어)
    ↓
[Phase 1] 의사 차트 → 구조 추출 (LLM)
    ↓
[Phase 2] 의사별 빈도/공출현 분석
    ↓
[Phase 3] 의사 본인 셀프 큐레이션 (30분~2시간)
    ↓
[Phase 4] 의사 전용 룰북 JSON 생성 + UI 로딩
    ↓
[Phase 5] 클릭 로그 누적 → 의사별 자동 갱신
```

이전 문서의 5단계와 골격은 같다. 차이점:

- **Phase 0**: 공통 vocabulary 외에 의사별 약어 사전을 점진 추가
- **Phase 3**: **의사 본인이 큐레이터**. 외부 의사 검토자가 아님
  - UX가 운영의 핵심 (마찰 최소화 + 품질 보장)
  - 큐레이션 인터페이스가 별도 제품 기능이 됨
- **Phase 5**: 의사별 클릭 로그 → 그 의사 룰북만 갱신

## 6. 콜드 스타트 처리 ★ 가장 중요한 신규 항목

차트가 적은 의사 (신규 개원, 전공의)에 대한 정책. 이 정책 없이는 시장 일부에서 가치 제안이 무너진다.

### 차트 양에 따른 단계 정책

| 차트 수 | 마이닝 가능성 | 제공 방식 |
|---|---|---|
| < 100 | 거의 불가 | **공통 베이스 템플릿만** (CCF 스키마 그대로). 추후 누적되면 마이닝 시작 |
| 100~500 | 부분 가능 | 베이스 템플릿 + **의사 차트의 빈출 칩만 위에 얹음** (얇은 개인화) |
| 500~2000 | 의미 있음 | 마이닝 풀 적용, 단 **저빈도 항목은 베이스로 폴백** |
| 2000+ | 충분 | 본격 개인화. 대부분의 슬롯에 의사 패턴 우세 |
| 10000+ | 풍부 | 부위별 세분화까지 가능 |

### 적립형 모델
- 차트 적은 의사도 가입은 가능
- 사용하면서 누적되는 **자기 차트 + 클릭 로그**로 룰북이 매월/분기 자동 강화
- "처음엔 일반 템플릿이지만 시간이 갈수록 당신만의 시스템이 됩니다"라는 메시지

### 위험
- 신규 의사가 **첫인상에서 차별화를 못 느낌** → 이탈
- 완화: 가입 시점에 베이스 템플릿 자체의 완성도가 높아야 함. 마이닝은 **부가가치이지 핵심**이 아니어야 함

## 7. 개인정보·법적 고려사항 ★ 신규 핵심 영역

### 데이터 흐름
```
의사 EMR → 우리 서버 → (LLM API) → 우리 서버 → CCF UI
        ↑              ↑              ↑
        ① 수집         ② 처리         ③ 저장/서빙
```

각 단계에 PHI 처리 규제 적용.

### 검토 필요 법령 (한국 기준)
- **의료법 시행규칙** (전자의무기록 보안)
- **개인정보보호법** (PHI는 민감정보)
- **보건복지부 EMR 인증 표준**
- 환자 동의 없는 외부 처리 가능 범위
- 가명처리·비식별화 가이드라인

### 기술 선택에 미치는 영향

| 옵션 | 장점 | 단점 |
|---|---|---|
| 해외 클라우드 LLM (Anthropic, OpenAI) | 정확도 최고, 즉시 가용 | **국외 이전 이슈**, BAA류 계약 필요 |
| 국내 클라우드 LLM (네이버 등) | 데이터 국내 거주 | 정확도 미지수 |
| 온프레미스 오픈소스 LLM | 데이터 외부 안 나감 | 인프라 비용 ↑, 정확도 ↓ |
| **비식별화 후 외부 LLM** | 절충안, 현실적 권장 | 비식별화 자체가 신뢰성 검증 필요 |

> **권장**: 비식별화 파이프라인 + 외부 LLM. 환자명·식별번호·주소·전화번호 제거 후 마이닝. 단, 차트 본문에 이름이 남는 경우(예: "동생이 한양아파트") 대비한 NER 비식별화 필요.

### 데이터 보유·삭제
- 의사 해지 시 룰북 + 원본 차트 즉시 삭제 가능 구조 필요
- 클리닉 단위 폐업 대응
- 환자별 삭제 요청 대응 (개인정보보호법)

## 8. 의사당 비용 구조

### 변동비 (의사 한 명당)
| 항목 | 추정 (의사 평균 기준) |
|---|---|
| LLM 호출 (Phase 1 추출) | 12,000건 × $0.002 = **약 3만원** (Haiku) ~ 약 12만원 (Sonnet) |
| 의사 큐레이션 시간 | 0.5~2시간 (자가 시간이지만 도입 마찰 비용) |
| 스토리지 (룰북 + 원본) | 무시 가능 |
| 정기 갱신 (분기별) | 약 3~5만원 |
| 고객 지원 | 변동 |
| **연간 의사당 변동비** | **15~30만원 추정** (LLM 모델 선택과 갱신 주기에 좌우) |

### 고정비
- 마이닝 파이프라인 개발/유지
- 비식별화 인프라
- PHI 컴플라이언스 인프라 (인증, 감사 로그)
- UI 큐레이션 도구

### 가격 시사점
- 의사당 변동비 ~30만원/년이라면, 손익분기는 구독료 + 셋업비 합 **연 100만원 이상** 필요 (마진 + 고정비 분담)
- 정형외과 외래 의사 평균 매출 대비 ROI 가능한가? → **시간 절감을 가격으로 정당화**해야 함 (예: 환자당 30초 절감 × 일 50명 × 연 250일 = 연 100시간 절감 → 의사 시간 시급 환산)

## 10. 장점 (냉정하게)

### 10-1. 차별화된 가치 제안
- 일반 EMR 템플릿과 명확히 구분되는 **"내 차트 그대로"** 메시지
- 의사 입장에서 직관적이고 즉시 이해 가능

### 10-2. 책임 소재 명확
- 추천이 의사 본인 패턴에서 나옴 → "왜 이걸 추천했는지" 설명 가능
- 의료 사고 시에도 의사 본인이 만든 패턴이라는 근거 → 외부 룰북 대비 법적 부담 ↓

### 10-3. 락인(switching cost) 자연 형성
- 의사 본인의 룰북은 시간이 갈수록 정교해짐 → 이탈 비용 ↑
- 클릭 로그가 누적되면 다른 시스템으로 못 옮김

### 10-4. 큐레이션 부담 분산
- 이전 모델: 외부 의사 한 명이 모든 룰을 큐레이션 (병목)
- 지금: 각 의사가 자기 룰만 큐레이션 (병렬)

### 10-5. 클리닉 단위 확장 가능
- 한 의사 도입 후 동료에게 추천 → 클리닉 단위 확산
- 원장 의사 도입 시 부원장·페이닥터 추가 가입 유도

### 10-6. 시간 경과 가치 증가
- 매일 차팅 = 데이터 누적 = 룰북 정교화
- 1년 사용한 의사의 룰북 > 신규 의사 룰북. 자연스러운 신뢰 누적

### 10-7. 기술 단순성
- 학습 모델 없음, 정적 JSON 배포
- 추론 비용 무시 가능 (룰 lookup만)
- 모델 업데이트 주기 = 마이닝 갱신 주기 (분기 단위)

### 10-8. 점진적 ML 강화 경로
- 의사별 클릭 로그 누적 → 마르코프/n-gram/트랜스포머로 점진 업그레이드
- 룰북은 폴백/안전망으로 영원히 유효

## 11. 단점 (냉정하게)

### 11-1. 콜드 스타트 ★
- 신규 개원의·전공의 시장에서 핵심 가치(개인화)가 작동 안 함
- 베이스 템플릿으로 회피 가능하지만, 그건 일반 EMR과 차별화 약함
- **신규 의사 시장을 포기하거나, 베이스 템플릿 자체의 완성도로 승부해야 함**

### 11-2. 데이터 수급 마찰
- 의사 본인 차트가 EMR(병원/클리닉) 시스템에 묶여 있음
- export 절차가 행정·기술적으로 길고 EMR 벤더 별로 다름
- 출시 병목이 **기술이 아니라 운영/영업**이 됨

### 11-3. PHI 처리 부담
- 의료법 컴플라이언스가 단순 SaaS보다 훨씬 무거움
- 비식별화·국외 이전·환자 동의 등 검토 항목 많음
- 컴플라이언스 인프라 자체가 일반 SaaS 대비 6~12개월 추가 개발 필요

### 11-4. 의사당 변동비
- LLM 호출이 의사당 비용을 끌어올림
- 가입 후 이탈 시 비용 회수 불가 → **무료 체험 정책 위험**
- 마진 압박 → 가격 인상 또는 LLM 다운그레이드 압박

### 11-5. 차트 품질 편차
- 풍부하게 적는 의사 vs 짧게 적는 의사 → 룰북 품질 천차만별
- 짧게 적는 의사가 "다른 의사보다 못한 결과"를 받으면 만족도 ↓
- 마이닝 품질에 의사 자신의 과거 행동이 영향 → 통제 어려움

### 11-6. Survivorship bias 강화
- 의사가 **고려했지만 채택 안 한** 진단/처방은 룰북에 없음
- 룰북이 의사의 기존 패턴만 강화 → 학습/발전 기회 차단 위험
- 완화: 베이스 템플릿을 항상 폴백으로 띄우기, "이 의사 패턴 외" 별도 표시

### 11-7. 큐레이션 UX 양날의 검
- 길면 이탈, 짧으면 품질
- 의사 본인이 자기 패턴의 문제점을 못 보는 경우도 있음 (블라인드 스팟)
- "외부 임상 검토" 옵션 추가 필요할 수 있음 (비용·복잡도 ↑)

### 11-8. 전문과목 한정
- CCF는 정형외과 위주 → TAM 좁음
- 다른 과 확장 시 스키마·룰북 전부 새로 짜야 함
- 정형외과 한국 개원의 시장 약 4,000명 → 100% 도입해도 매출 천장 명확

### 11-9. 데이터 누적 곧 의사 이직 시 분쟁
- 룰북이 의사 개인 자산인지 클리닉 자산인지 모호
- 클리닉 단위 도입 후 의사 이직 시 룰북 소유권 다툼 가능
- 계약 조항 사전 정리 필요 (법무 비용)

## 12. 사업성 (냉정하게)

### 12-1. 가격 모델 옵션

| 모델 | 장점 | 단점 |
|---|---|---|
| **월 구독 단일가** | 단순, 예측 가능 | 마이닝 변동비 회수 어려움 |
| **셋업비 + 월 구독** | 변동비 회수 명확 | 도입 장벽 ↑ |
| **사용량 과금** (환자 수 비례) | 가치-가격 정합 | 의사 입장에서 비용 예측 어려움 |
| **클리닉 단위 패키지** | LTV ↑ | 영업 사이클 길어짐 |

> **현실적 권장**: 셋업비(~50만원) + 월 구독(~5~15만원) 조합. 셋업비는 마이닝 변동비 + 도입 진정성 필터링.

### 12-2. 단위 경제

매우 거친 추정 (실제 검증 필요):

```
의사당 LTV (3년 가정):
  연 구독 120만원 × 3년 = 360만원
  - 셋업비 50만원 (1회)
  = 410만원

의사당 비용 (3년):
  - 변동비: 30만원 × 3년 = 90만원
  - 영업/획득(CAC): 100~300만원 (B2B 의료 SaaS 통상)
  - 지원: 50만원 × 3년 = 150만원
  = 340~540만원

→ 단위 마진: -130 ~ +70만원/3년
```

CAC가 100만원 이하로 떨어지지 않으면 **단위 경제 자체가 위태롭다**. 해법:
- 클리닉 단위 도입으로 CAC 분산
- 의사 추천 프로그램으로 CAC 절감
- 셋업비 인상 또는 LLM 다운그레이드로 변동비 절감

### 12-3. 시장 규모 (TAM)

- 한국 정형외과 개원의: 약 4,000명
- 봉직의 포함 약 8,000명
- **현실적 SAM** (개원의 도입 가능 비율): 30~50% = 1,200~2,000명
- **현실적 ARR 천장**: 의사당 연 120만원 × 1,500명 = **약 18억원/년**

해외 진출이나 인접 과(통증의학, 재활의학) 확장 없이는 IPO급 회사 어려움. **틈새 SaaS 또는 인수 출구**가 더 현실적 시나리오.

### 12-4. 경쟁 위협

| 위협 | 시기 | 대응 |
|---|---|---|
| 기존 EMR 벤더가 같은 기능 탑재 | 1~2년 | 데이터 + 큐레이션 자산을 빠르게 누적, 락인 강화 |
| 음성 인식 차팅 (Nuance류) | 진행 중 | CCF는 구조화 + 보험청구 친화 메시지로 차별화 |
| AI 자동 차팅 (LLM 직접) | 2~3년 | 책임 소재 명확성과 의사 통제권으로 차별화 |
| 한국형 EMR 인증 강화 | 1년 이내 | 컴플라이언스 인프라 선제 구축 |

### 12-5. 사업성 종합 평가

- **틈새 SaaS로는 성립 가능** — 단위 경제 마진 박하지만 정형 시장에서 인지도 확보 가능
- **빅 시장 기대 어려움** — 단일 전문과 + 한국 한정 + 단위 경제 박함
- **인수 출구가 합리적 종착점** — 기존 EMR 벤더 또는 대형 SaaS 인수
- **핵심 위험**: CAC, 컴플라이언스, EMR 벤더 모방
- **핵심 기회**: 의사별 데이터 누적이 시간 지나며 해자가 됨 (선점 효과)

## 13. 구현 가능성 (냉정하게)

### 13-1. 기술적 구현 (가능)
- 마이닝 파이프라인: LLM 활용 시 충분히 가능. 6주 안에 1 의사 MVP
- 비식별화: 한국어 의료 NER 모델이 부족하지만 LLM으로 대체 가능
- 룰북 서빙: 단순 정적 JSON, 표준 SaaS 스택
- 클릭 로그: 표준 분석 파이프라인 (Mixpanel/Amplitude/자체)
- **기술적 난이도: 중**

### 13-2. 운영적 구현 (어려움)
- 의사별 데이터 export: EMR 벤더마다 절차/포맷 다름. 자동화 어려움
- 도입 첫 1~2주가 사람이 붙어야 하는 작업 → 스케일 한계
- 의사 큐레이션 UX: 반복 개선 필요. 첫 버전은 만족도 낮을 가능성
- **운영적 난이도: 중상**

### 13-3. 법적 컴플라이언스 (가장 어려움)
- 의료법, 개인정보보호법, EMR 인증 표준 각각 검토
- 비식별화 파이프라인 신뢰성 검증 (감사 가능 형태로)
- LLM API 외부 호출 = 국외 이전 = 환자 동의 + 처리 위탁 계약
- 의료기관 인증 또는 의료기기로 분류될 가능성 사전 자문 필수
- **법적 난이도: 상**

### 13-4. 현실적 타임라인

| 단계 | 기간 | 산출물 |
|---|---|---|
| 1 의사 PoC (수동 마이닝) | 6주 | 시연용 MVP |
| 자동화 + 컴플라이언스 1차 | +6개월 | 알파 (소수 의사 클로즈드 베타) |
| 비식별화·감사 인프라 | +3개월 | 베타 (10~50 의사) |
| 정식 출시 | +3개월 | 일반 가입 가능 |
| 100 의사 확보 | +6~12개월 | 단위 경제 검증 가능 |

총 **18~24개월에 100 의사 확보** 시나리오. 이게 낙관 시나리오.

### 13-5. 가장 큰 구현 리스크
- **EMR 데이터 export 자동화 실패** → 의사 손이 많이 가서 도입율 ↓
- **컴플라이언스 검토 결과 LLM 외부 호출 불가 판정** → 온프레미스 LLM 강제 → 비용/정확도 양쪽 타격
- **큐레이션 UX 첫 버전 실패** → 도입 의사 만족도 ↓ → 추천 끊김

## 14. 예상 정확도 (냉정하게)

정확도는 단계별로 다르며, 각 단계의 오차가 누적된다.

### 14-1. Phase 1 (자유 텍스트 → 구조 추출)
- LLM 사용 시 **85~95%** (Sonnet/Haiku)
- 정규식만 사용 시 **60~70%**
- 의사 약어 정규화 후 추가 5%p 개선 가능
- **권장 추정치: 90%**

### 14-2. Phase 2 (마이닝 후 룰북 생성)
- Phase 1 정확도가 그대로 빈도 노이즈로 전이
- 그러나 빈도 임계값(≥5건) 적용 시 노이즈 대부분 필터링
- 의사 큐레이션이 마지막 정제
- **사용자가 보는 룰북 자체의 품질: 90~95%**

### 14-3. Phase 4 (UI에서 다음 칩 추천)

| 지표 | 차트 5,000+ 의사 | 차트 1,000~5,000 | 차트 <500 (콜드 스타트) |
|---|---|---|---|
| **칩 커버리지** (정답이 후보 안에 있는 비율) | 85~90% | 70~80% | 50~65% (베이스 템플릿 의존) |
| **top-1 추천 정확도** | 40~55% | 30~45% | 25~35% |
| **top-3 추천 정확도** | 70~85% | 55~70% | 45~60% |
| **자유 입력 비율** | 15~20% | 25~30% | 35~50% |

### 14-4. 진단→처방 패키지 정확도
- CSV의 `진단목록 × 처방목록`은 깔끔한 구조 → 마이닝 정확도가 자유 텍스트보다 높음
- **top-1: 50~70%, top-3: 80~92%**

### 14-5. 실제 가치 지표 (시간 절감)
- 차트당 입력 시간 단축: **20~40%** (차트 풍부한 의사일수록 ↑)
- 콜드 스타트 의사: 5~15% (사실상 베이스 템플릿 효과)
- 6개월 사용 후: +10%p 추가 절감 예상 (클릭 로그로 룰북 강화)

### 14-6. 정확도 종합 판단
- **충분히 유용함**: top-3 70~80%면 사용자 만족 임계값 통과
- **하지만 콜드 스타트 의사에게는 가치 약함**: 별도 메시징·정책 필요
- **첫 1년이 결정적**: 클릭 로그 누적되면 정확도 개선 가속화

> 위 수치는 **추정치**다. 실측 검증 전엔 마케팅 자료에 사용하면 안 됨.

## 16. 합의 필요 사항 (개정)

1. **콜드 스타트 정책**: 차트 N건 미만 의사 → 베이스 템플릿만? 가입 거절? 적립형?

3. **가격 모델**: 셋업비 + 구독 vs 단일 구독 vs 사용량 과금
4. **큐레이션 UX 시간 예산**: 의사 첫 세션에 몇 분?
5. **룰북 소유권**: 개인 vs 클리닉 (계약서 조항 사전 정리)
6. **EMR 통합 우선순위**: 어떤 EMR 3~5개를 먼저 지원할지
7. **MVP 범위**: 1 부위(허리)부터 vs 전 부위 동시
8. **출시 후 클릭 로그 인프라 도입 시점**

## 18. 결론 (한 줄)

**의사별 마이닝 온보딩은** 범용 룰북보다 **가치 제안이 명확하고 락인이 자연스럽지만**, 콜드 스타트·데이터 수급·컴플라이언스·단위 경제 같은 **운영/사업 측면의 새로운 한계**를 동반한다. 기술적 구현은 가능하고 정확도도 충분하나, **진짜 승부는 영업·법무·UX**에서 갈린다. 18~24개월 선행 투자를 감내할 수 있는 팀이 가져야 할 모델이다.
