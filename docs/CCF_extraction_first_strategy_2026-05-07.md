# CCF 전략 합성 — 클릭 차트의 진짜 목적과 표준 구조화 데이터의 가치

> 작성일: 2026-05-07
> 작성 맥락: 일주일에 걸친 일련의 전략 검토(`CCF_strategic_decision`, `CCF_ai_recommendation_assessment`, `CCF_mining_plan` 등) 끝에 도달한 합성 문서
> 목적: 그동안의 가정을 해체하고, **클릭 차트의 진짜 목적**과 **표준 구조화 데이터로 가능한 것**을 정리

---

## 0. 이 문서를 쓰는 이유

지난 며칠간 우리는 다음 흐름으로 검토를 진행했다:

1. CCF 클릭 차트 SOAP 스키마 정의
2. 12,284건 데이터로 의사 패턴 마이닝 → 마르코프 체인 검토 → 룰북 큐레이션 검토
3. 의사별 온보딩 관점으로 마이닝 재정의
4. 자유 텍스트 + AI 추천(검색형) 제품 검토
5. 클릭 컨텍스트 기반 ICD 추천(B안) 검토
6. 클릭 차트의 NLP 난이도 재평가
7. **결정적 성찰**: 클릭 차트는 자유 표현을 막는다. 진짜 목적은 무엇이었나?
8. 자유 텍스트만으로는 안 됨 — 표준어 매핑이 필요

이 문서는 7~8번 통찰을 진지하게 받아들여, **그동안 각 문서에서 부분적으로 다루던 가정들을 통합 해체**하고, **새로운 기준점을 제시**한다. 이전 문서들의 가치를 부정하지 않되, 그 위치를 재정의한다.

## 1부 · 가정의 해체 — 우리가 잘못 놓고 있던 질문

### 1.1 출발점은 어디였나

CCF는 본래 **"의사가 차트를 더 빠르게 쓰게 하자"** 에서 출발했다. 자연스러운 설계가 따랐다:

- SOAP를 부위·항목으로 분해 → 클릭 칩으로 미리 만들어두기
- 의사가 직접 타이핑할 일 없도록 → 클릭으로 입력
- 이 입력이 곧 **구조화된 데이터** → 데이터베이스 저장 가능

이게 **CCF 슬라이드 PDF**(`CCF(Create Chart Faster)`)의 본 그림이었고, 우리가 받아들인 전제였다.

### 1.2 첫 균열 — UI 복잡도

스키마를 그대로 UI로 옮겨보면, **의사가 한 화면에 마주해야 하는 항목 수가 급격히 늘어난다.**
무릎 하나만 봐도:
- 측 (좌/우)
- 세부부위 (내측/외측/앞쪽/뒤쪽)
- 구조물 (Meniscus, MCL, Pes anserius, VMO, MPFL …)
- 발병 시기, 양상, VAS, 악화 요인, 완화 요인 …

부위마다 이걸 곱하면 클릭 칩이 수백 개가 된다. UI 단순화의 한계에 부딪힌다.

### 1.3 두 번째 시도 — 개인화로 칩 줄이기

그래서 우리는 **개인화**로 우회했다:
- 의사가 자주 쓰는 칩만 위로
- 다음에 누를 칩을 추천 (반응형)
- 그러려면 의사 패턴 학습 필요 → 12,284건 마이닝
- 마르코프 체인, 룰북 큐레이션 등 검토

여기까지가 `CCF_mining_plan.md`, `CCF_charting_markov_design.md`, `CCF_SOAP_case_decomposition.md`의 문맥이었다.

### 1.4 결정적 질문

그러던 중 사용자가 제기했다:

> "근데 클릭 차팅을 결국에 왜 하지? 잘못하면 자유로운 표현을 막는 거 아닌가?
> 결국 클릭 차팅은 구조화된 데이터로 저장하여 환자의 유사 사례나 마케팅,
> 그리고 차팅 시 빠뜨린 부분 등을 잡아낼 수 있게 하기 위해 고민했던 게 아닌가?"

이 질문이 결정적이었다. 우리가 그동안 놓치고 있던 것을 정확히 짚는다.

### 1.5 잘못 놓고 있던 질문은 무엇이었나

우리는 줄곧 다음 질문을 풀고 있었다:

> **"어떻게 하면 클릭 입력을 사용성 있게 만들까?"**

하지만 진짜 질문은 이거였어야 한다:

> **"왜 구조화된 데이터가 필요한가? 입력 방식과 그 목적은 분리 가능한가?"**

전자에 답하면 영원한 UI 최적화 게임이 된다. 후자에 답하면 **입력 UX와 데이터 구조화를 분리**할 길이 보인다.

## 2부 · 클릭 차트의 진짜 목적 — 30년 의료 정보학의 미해결 문제

### 2.1 EMR의 역사적 진동

전자의무기록(EMR)은 1990년대부터 **두 가지 모순된 요구**를 동시에 만족시키려 했다:

| 차원 | 의사가 원하는 것 | 시스템이 원하는 것 |
|---|---|---|
| 표현 | **자유 텍스트** (풍부한 뉘앙스, 빠른 입력) | **구조화 입력** (표준 어휘, 비교 가능) |
| 입력 시간 | 짧을수록 좋음 | 어떻게든 표준화 |
| 정확성 | 의사 머릿속이 곧 진실 | 코드화 가능해야 함 |

이 둘은 **물리적으로 양립 불가능**해 보였다. 그래서 EMR은 두 진영을 진동했다:

- **자유 텍스트 우선 EMR**: 의사 만족도 ↑, 다운스트림 데이터 활용도 ↓
- **구조화 입력 강제 EMR**: 데이터 활용도 ↑, 의사 만족도 ↓ (이탈, 외부 EMR로 도주)

한국에서 이 진동의 결과가 곧 현재의 EMR 환경이다. 대부분의 진료 차트는 자유 텍스트 + 진단 코드/처방 코드만 표준화된 형태로 남는다. 그 사이의 풍부한 정보(검진 결과, 증상, 시기, 치료 반응)는 자유 텍스트에 녹아 있어 후처리가 어렵다.

### 2.2 클릭 차트는 무엇을 시도했나

CCF의 클릭 차트는 본질적으로 이 30년 문제에 대한 한 시도였다:

> "**의사에게 친숙한 칩 UI를 만들면**, 자유 텍스트보다 빠를 수도 있고, 그러면서도 구조화 데이터가 떨어진다. 양립 가능하지 않을까?"

이게 CCF의 본 가설이었다. 그리고 이 가설의 핵심 가치는 **"입력 속도"가 아니라 "표준화된 데이터의 산물"**이다.

다시 말해, **클릭 차트의 본질은 입력 UI가 아니다. 입력 UI는 수단이었다.** 본질은:

- **표준 어휘**: 의사가 다양하게 표현하는 것을 통일된 용어로 묶기
- **구조화 저장**: 데이터를 후속 분석·검색·매칭 가능한 형태로 저장
- **코드 매핑**: 진단·처방을 보험·통계·연구에 쓸 수 있도록

이 세 가지가 진짜 목적이었고, 클릭 UI는 그것을 달성하기 위한 한 가지 방법이었다.

### 2.3 그러나 클릭 UI는 항상 사용성에서 진다

이게 의료 정보학에서 30년간 반복된 패턴이다. 왜?

1. **자유 텍스트는 의사의 인지 흐름과 일치**: 진찰하면서 떠오르는 순서대로 적음. 클릭은 **카테고리 → 항목 → 값**으로 의사를 강제로 분기시킴.
2. **표현의 풍부함을 잃음**: "예전엔 잘 쓰던 말이 칩에 없네" → 자유 입력으로 도주 → 결국 텍스트로 회귀.
3. **간헐적 예외 케이스**: 일반적이지 않은 환자는 클릭 트리에 없는 항목을 요구. 매번 자유 텍스트로 도주.
4. **인지 부담 분산**: 자유 텍스트는 흐름 한 번. 클릭은 매 항목마다 "찾고 → 선택" 두 단계.

결과: **의사들은 자유 텍스트를 이긴다. 늘.** 클릭 UI를 강제하면 EMR을 이탈한다. 한국에서도 미국에서도 같은 패턴이 반복된다.

### 2.4 그럼에도 표준화된 데이터는 반드시 필요하다

여기에 진짜 모순이 있다. 의사는 자유 텍스트를 원하지만, **자유 텍스트만으로는 의료 시스템이 굴러가지 않는다.**
- 보험 청구는 ICD/처방 코드 필요
- 통계는 표준 진단명 필요
- 검색은 비교 가능한 데이터 필요
- 연구는 코호트 추출 필요
- 마케팅은 환자 세분화 필요

그래서 EMR은 늘 **타협**한다. 진단 코드와 처방 코드만 강제 입력시키고, 나머지는 자유 텍스트로 받는다. 이게 한국 EMR의 현 상태다.

이 타협의 부작용: **자유 텍스트 안에 갇힌 풍부한 정보가 활용되지 못한다.** 검진 결과, 증상의 정도, 치료 반응, 환자 호소 — 다 자유 텍스트로만 존재.

### 2.5 본질의 재정의

이 모든 것을 정리하면:

> **클릭 차트의 진짜 목적은 "자유 텍스트 안에 갇힌 정보를 표준 어휘로 끌어내기"였다.**
> 입력 UI는 그것을 달성하기 위한 한 가지 도구일 뿐, 본질이 아니다.
> 그리고 그 도구는 **30년간 사용성에서 졌다.**

따라서 진짜 질문은 이렇게 바뀌어야 한다:

> **"의사의 자유 텍스트 입력을 막지 않으면서, 표준 어휘로 구조화된 데이터를 어떻게 얻을 것인가?"**

## 3부 · 지금 무엇이 바뀌었나 — LLM이라는 새 변수

### 3.1 30년 묵은 모순의 해결 가능성

위 질문은 30년간 풀리지 않았다. 이유는 단순하다: **자유 텍스트를 표준 어휘로 매핑하는 NLP의 정확도가 부족했다.**

- 1990년대 키워드 매칭: 30~50% 정확도
- 2000년대 정규식 + 의료 사전: 50~70%
- 2010년대 통계 NER: 65~80%
- 2020년대 BERT 기반 의료 NLP: 75~85%
- **2024년+ LLM (Claude/GPT)**: **85~95%**

마지막 도약이 결정적이다. 90% 이상의 매핑 정확도는 처음으로 **운영 가능한 수준**이다.

### 3.2 비용도 같이 떨어졌다

- 2020년 GPT-3 추출: encounter당 $0.10~0.50
- 2024년 Haiku/Sonnet: encounter당 **$0.005~0.05** (1/10~1/100)
- 프롬프트 캐싱: 추가 50~80% 절감

### 3.3 새로운 가능성

이 두 변수(정확도 90%+ × 비용 운영 가능)가 처음으로 만나면서, 30년 모순에 대한 실제 해결책이 가능해진다:

```
[입력]  의사 자유 텍스트 (지금처럼)
            ↓ (의사는 모름, 백그라운드)
[추출]  LLM이 표준 어휘로 매핑
            ↓
[저장]  구조화된 데이터로 저장
            ↓
[활용]  다양한 다운스트림 제품
```

의사 경험 변화: **0**.
시스템 활용도 변화: **자유 텍스트 EMR 대비 10배**.

### 3.4 이건 단순 LLM 기술 도입이 아니다

핵심은 LLM 사용 자체가 아니라, **"입력 UX와 데이터 구조화를 분리할 수 있게 됐다"**는 것이다. 이건 의료 정보학에서 패러다임 전환에 해당한다.

## 4부 · 표준어 구조화 데이터로 무엇을 할 수 있는가

본문의 핵심. 사용자가 묻는 질문이다. 이해관계자별로 분류한다.

### 4.1 의사 본인 (진료 중 + 진료 후)

**진료 중 가치**

| Use Case | 기능 | 의사가 느끼는 가치 |
|---|---|---|
| 차팅 누락 알림 | "HNP 의심인데 SLR 검진 안 적으셨어요" | 의료 사고 예방, 청구 거절 방지 |
| 유사 사례 검색 | "같은 부위·진단·검사 결과 환자 5건 — 이렇게 치료했었음" | 의사결정 보조, 자신감 |
| 진단 보조 | "이 증상 조합엔 OA보다 ChMP가 더 흔합니다" | 진단 정확도, 학습 |
| 처방 패턴 자가 모니터 | "오늘 처방이 본인 평소 패턴과 다릅니다" | 일관성, 자가 audit |
| 약물 상호작용 경고 | 진단 + 복용 약 간 충돌 검사 | 환자 안전 |
| 진료 시간 예측 | 호소 패턴별 평균 진료 시간 | 일정 관리 |

**진료 후 가치**

| Use Case | 기능 | 의사가 느끼는 가치 |
|---|---|---|
| 본인 차팅 통계 | 진단 분포, 처방 패턴, 시간 추이 | 자기 이해, 성장 |
| 청구 적정성 self-audit | 진단 vs 처방 매칭 부적절 케이스 | 청구 거절 사전 방지 |
| 의료 가이드라인 준수 점검 | 표준 가이드 vs 본인 패턴 | 임상 품질 향상 |
| 의료 학회 발표용 데이터 | 비식별 코호트 추출 | 학술 활동 지원 |

### 4.2 클리닉 운영자 (원장·사무장)

| Use Case | 기능 | 운영 가치 |
|---|---|---|
| **마케팅 환자 타게팅** | "주사 치료 후 6개월 이내 미내원" + "특정 진단" 필터 | 재방문율 향상, 매출 직접 기여 |
| 청구 거절·삭감 사전 예방 | 진단-처방 매칭 자동 검증 | 청구 실수 → 손실 ↓ |
| 진료 통계 대시보드 | 일/주/월 진단 분포, 처방 패턴 | 운영 의사결정 |
| 의사별 패턴 비교 (다중 의사) | 동일 진단의 처방·검사 차이 | 임상 일관성, 신규 의사 트레이닝 |
| 환자 retention 분석 | 첫 방문 → N회 재방문 전환율 | 고객 이해, 마케팅 ROI |
| 보험사 청구 데이터 분석 | 보험사별 거절률, 삭감 패턴 | 청구 전략 최적화 |
| 주사·시술 수요 예측 | 시즌별 처방 패턴 | 재고·인력 관리 |

### 4.3 환자 본인 (선택적 노출)

| Use Case | 기능 | 환자 가치 |
|---|---|---|
| 진료 결과 요약 | 자유 텍스트 차트 → 표준 요약 | 이해, 보관 |
| 처방 이력 | 시간 순 처방 모아보기 | 약물 관리 |
| 다른 의사에게 이전 시 | 표준 형식 의무기록 export | 의료 연속성 |
| 진단·치료 교육 자료 | 표준 진단명 → 일반인 설명 | 환자 교육 |

### 4.4 의료기관 단위 (병원·체인)

| Use Case | 기능 | 기관 가치 |
|---|---|---|
| Quality Indicator 측정 | 진단별 표준 검사 수행률 | 인증 평가, 평판 |
| 의사 간 처방 비교 | 동일 진단 처방 다양성 | 의국 운영 |
| 코호트 추출 (연구) | "허리 디스크 + 50대 + 1년 이상 추적" | 임상 연구 가속 |
| 신규 의사 패턴 학습 | 선임 의사 차팅 → 학습 보조 | 트레이닝 |
| 약물 부작용 모니터링 | 처방 후 증상 변화 패턴 | 환자 안전 |
| 임상 가이드라인 준수도 | 표준 가이드 vs 실제 패턴 | 의료 품질 |

### 4.5 산업 단위 (대규모 데이터 활용)

| Use Case | 기능 | 산업 가치 |
|---|---|---|
| **Real-world evidence** | 약물·시술의 실세계 효과 측정 | 제약사·식약처 의사결정 |
| 비식별 의료 데이터셋 | 연구·기업에 라이선스 | 직접 매출 |
| 새로운 임상 패턴 발견 | 패턴 마이닝 → 가설 생성 | 의료 발전 기여 |
| 의료 AI 모델 학습 데이터 | 표준화된 코호트 | AI 산업 기여 |
| 보험사 인수 보조 | 진단·처방 패턴 → 위험 평가 | 보험 산업 |
| 보건당국 정책 자료 | 진단 분포 추이, 지역별 차이 | 정책 결정 |

### 4.6 이 모든 가치의 공통 전제

위 50개 가까운 use case를 가능하게 하는 단 하나의 조건:

> **"표준 어휘로 매핑된 구조화된 데이터가 충분한 양으로 누적되어야 한다."**

자유 텍스트 EMR은 이 조건을 못 만든다. 클릭 강제 EMR은 사용성 때문에 데이터 양을 못 만든다. **자유 텍스트 + 자동 추출**이 처음으로 두 조건(자연스러운 입력 → 양 + 표준 매핑 → 구조)을 동시에 만족시킨다.

## 5부 · 새 시스템 아키텍처

### 5.1 5-Layer 모델

```
┌─────────────────────────────────────────────────────┐
│ Layer 5 — 다운스트림 가치 제품들                       │
│  마케팅 / 누락 알림 / 유사 사례 / 청구 검증 /          │
│  통계 대시보드 / 코호트 / 환자 요약 ...                │
├─────────────────────────────────────────────────────┤
│ Layer 4 — 구조화 저장소                              │
│  표준 어휘로 매핑된 SOAP 데이터                       │
│  (DB 또는 데이터 웨어하우스)                          │
├─────────────────────────────────────────────────────┤
│ Layer 3 — 검증·보정 UI (선택적)                      │
│  AI 추출 결과를 의사가 확인·수정                       │
│  (옛 클릭 차트의 자리)                                │
├─────────────────────────────────────────────────────┤
│ Layer 2 — 추출 엔진                                  │
│  LLM (또는 NER) 기반                                 │
│  자유 텍스트 → CCF 온톨로지 매핑                      │
├─────────────────────────────────────────────────────┤
│ Layer 1 — 자유 텍스트 입력                           │
│  타이핑 또는 음성 → 텍스트                            │
│  (의사 경험 변화 0)                                   │
└─────────────────────────────────────────────────────┘

[Layer 0] 어휘 시스템 (이중 구조)
  ├─ 표준 어휘: CCF 스키마 + KCD-8 + KIMS + 심평원 코드
  │   (부위·진단·처방·시술 등 finite·외부 표준)
  └─ 의미 클러스터: 임베딩 기반 자동 군집
      (통증 양상·서술 표현 등 다양성 보존 필요)
  (모든 layer의 공통 참조 — §7.3 참조)
```

### 5.2 각 Layer의 책임

| Layer | 책임 | 기술 선택 |
|---|---|---|
| 0 | 어휘 시스템 (표준 + 클러스터) | CCF 스키마 + 외부 표준(KCD-8/KIMS/심평원) + 임베딩 클러스터링 (§7.3) |
| 1 | 자유 텍스트 입력 받기 | 음성 인식 + 텍스트 에디터 |
| 2 | 추출 매핑 | LLM (Claude/GPT/Haiku) + 규칙 보조 |
| 3 | 의사 검증 UI | 가벼운 카드형 UI (강제 X, 알림성) |
| 4 | 구조화 저장 | PostgreSQL + 인덱싱 |
| 5 | 가치 제품들 | 각 제품 별도 |

### 5.3 의사 경험 흐름

```
의사: 평소처럼 자유 텍스트 입력 (변화 0)
       ↓
시스템: 백그라운드에서 추출 시작
       ↓
   [매핑 신뢰도 높음]    [매핑 신뢰도 낮음]
        ↓                       ↓
   조용히 저장          의사에게 카드 알림
   (의사 모름)         "이거 맞나요? 수정?"
                              ↓
                         의사 1초 확인 클릭
                              ↓
                         학습 데이터로 저장
```

핵심: **의사는 평소대로 일하고, 시스템은 백그라운드에서 데이터를 만든다.**
강제 클릭 차트의 30년 함정에서 빠져나오는 길이다.

## 7부 · 정직한 한계

### 7.1 한국어 의료 NLP의 빈약함

LLM이 의료 NLP 정확도 90% 도약을 가능하게 했지만, 이는 **영어 의료 텍스트 기준**이다.

한국어 의료 텍스트는:
- 약어가 많음 (한자 + 영문 약어 + 의사 개인 약어)
- 구어체와 문어체가 섞임 ("환자가 그랬다" + "Acute lumbar sprain")
- 표준 의료 용어집(SNOMED-CT 한국어 버전)이 빈약
- 한국어 의료 NER 학습 데이터셋 부족

→ **GPT/Claude의 정확도가 영어 대비 5~10%p 낮을 가능성**. 90% → 80~85%.

대응:
- 정형외과 도메인 한정 → 어휘 좁힘 → 정확도 보전
- Few-shot prompting + RAG (의사별 약어 사전)
- 의사 검증 UI(Layer 3)로 부족분 보충

### 7.2 매핑 정확도의 비대칭

**모든 매핑 오류가 같은 비용은 아니다:**

| 오류 유형 | 비용 |
|---|---|
| 진단 코드 오추출 | **높음** (청구·통계·법적 문제) |
| 처방 약물명 오추출 | **매우 높음** (안전 문제) |
| 검진 결과 오추출 | 중간 (진료 결정에 영향) |
| 호소 부위 오추출 | 낮음 (의사가 곧 보정) |

→ 매핑 시스템은 **균일한 정확도가 아니라 카테고리별 임계값**을 가져야 함.
중요 카테고리는 의사 검증을 강제, 덜 중요한 것은 자동 통과.

### 7.3 표준화의 경계 — 무엇을 표준화하고 무엇을 클러스터링할 것인가

이전 초안에서는 "표준화의 정치"라며 모든 어휘 표준화를 동일 선상에 두고 비관적으로 평가했지만, **이건 잘못된 프레이밍**이었다. 항목 성격에 따라 처리 방식이 달라야 한다.

#### 7.3.1 두 부류의 항목

| 부류 | 처리 방식 | 이유 |
|---|---|---|
| **구조적·객관적 항목** | **표준화** (고정 어휘) | 외부 코드 시스템과 연동 필요, 비교·집계 불가피 |
| **서술적·주관적 항목** | **임베딩 클러스터링** | 표현 다양성 보존 필요, 강제 합치면 정보 손실 |

#### 7.3.2 표준화 대상 (고정 어휘)

이미 외부에 표준이 존재하거나, 본질적으로 유한한 항목들. 무리 없이 표준화 가능.

| 항목 | 표준 출처 |
|---|---|
| 부위 (anatomical site) | 해부학 자체가 finite (CCF 스키마 12부위 + 세부) |
| 진단 (diagnosis) | KCD-8 (한국 ICD-10) |
| 처방 약물 | KIMS DB |
| 시술·오더·검사 코드 | 심평원 청구 코드 |
| 검사 결과 | binary (양성/음성) 또는 numeric (각도, %, grade) |

이 항목들은 **외부 표준이 이미 답을 가지고 있다**. 우리가 새로 만들 필요 없음. 매핑만 하면 됨.

#### 7.3.3 클러스터링 대상 (의미 군집)

서술적 표현. 의사마다 다양한 어휘를 사용해도 의미가 비슷하면 같은 군집으로 자동 묶음.

| 항목 | 예시 표현 다양성 |
|---|---|
| 통증 양상 | 쑤시는 / 찌릿한 / 욱신거리는 / 뻐근한 / 화끈거리는 / 묵직한 |
| 발병 양상 | 갑자기 / 서서히 / 점차 심해지는 / 간헐적으로 |
| 악화·완화 표현 | 걸을 때 / 보행 시 / 보행하면 / 걷다가 |
| 환자 주관 호소 | "찌르는 것 같다" / "콕콕 쑤신다" / "벼락 맞은 것 같다" |

#### 7.3.4 클러스터링이 작동하는 메커니즘

```
의사 A: "쑤시는 통증"  ┐
의사 B: "찌릿한 통증"  ├─→  임베딩 거리 측정  ┐
의사 C: "콕콕 쑤심"    │                       ├─→  Cluster_pain_sharp_001
의사 D: "찌르는 느낌"  ┘                       │     (centroid: 날카로운 통증)
                                              │
의사 E: "둔한 통증"    ┐                       │
의사 F: "묵직한 느낌"  ├─→  임베딩 거리 측정  ┴─→  Cluster_pain_dull_002
의사 G: "뻐근함"       ┘
```

- **저장**: 원문 + 클러스터 ID 둘 다 보관 (원문은 표시용, 클러스터 ID는 검색·통계용)
- **검색·집계**: 클러스터 ID로 묶어서 처리 → "이 의사가 날카로운 통증 호소 환자에게 자주 처방한 것"
- **표시**: 의사 본인 차트 화면에는 원문 그대로 → 표현의 풍부함 유지
- **신규 표현**: 임베딩 거리로 기존 클러스터에 자동 편입 또는 새 클러스터 형성. **표준 어휘 갱신 회의 불필요.**

#### 7.3.5 경계가 모호한 항목 — 둘 다 허용

| 항목 | 처리 |
|---|---|
| "양측 무릎 통증" | 부위(표준화: 무릎/양측) + 호소(원문 보존) 동시 추출 |
| 흔치 않은 진단명 | KCD-8에 매핑 시도 → 실패 시 "미분류" 태그 + 원문 보존 |
| 의사 약어 (LBP, HNP) | 공통 약어 사전(표준화) + 의사별 약어(클러스터링 학습) |

→ 둘 중 하나로 강제하지 않고, **두 representation을 동시에 저장**한다. 다운스트림 제품이 필요한 쪽을 골라 사용.

#### 7.3.6 결론

"표준화의 정치"가 아니다. 표준화 대상과 클러스터링 대상의 **경계만 잘 그으면** 정치적 갈등은 거의 없다.

- **표준화는 외부 코드(KCD-8, KIMS, 심평원)의 매핑 문제** — 답이 이미 있음
- **클러스터링은 임베딩 + 자동 군집화** — 갱신 자동, 정보 손실 0
- **경계 항목은 이중 저장** — 강제 선택 없음

남는 운영 부담은:
- 클러스터 적정 입자 크기 튜닝 (너무 잘게/너무 거칠게)
- 신규 클러스터의 검토 주기 (분기 1회 정도)
- 외부 코드 갱신 추적 (KCD 개정, 심평원 코드 변경)

이건 "정치"가 아니라 **데이터 운영의 일상 작업**이다.

### 7.4 의사가 검증 UI를 무시할 위험

Layer 3 검증 UI는 의사가 봐야만 작동한다. 그런데:

- 매번 알림이 뜨면 피로 → 무시
- 첫 며칠은 봐도, 일주일 지나면 자동 클릭
- 무시되면 매핑 오류가 누적됨

→ 알림 빈도 자동 조정 필수. 신뢰도 95% 이상은 자동 저장, 80~95%만 알림, 80% 미만은 강하게 알림.

### 7.5 데이터 활용의 법·윤리

표준 구조화 데이터가 누적되면 강력한 자산이 된다. 동시에 강력한 위험도 된다.

- 의사별 패턴 분석 → 의사가 거부할 수도
- 환자 코호트 추출 → 동의 범위
- 보험사·제약사 데이터 판매 → 동의 + 비식별화 + 법적 검토
- 해킹 시 PHI 유출

→ 이건 **기술 전에 거버넌스가 선행**해야 한다. 데이터 거버넌스 정책이 없으면 어느 가치 제품도 출시 못 한다.

### 7.6 의사 수용성

가장 큰 한계. 백그라운드 추출이라 해도:
- "내 자유 텍스트가 AI에 분석된다"는 거부감
- 의사들의 변화 저항
- 추출 결과가 자기 의도와 다를 때 감정적 반응

→ **명시적 동의 + 끄기 옵션 + 단계적 노출** 필수. 강제하면 EMR을 떠난다.

**목표**: LLM 기반 자유 텍스트 → CCF 온톨로지 매핑 정확도 측정

작업:
- 12,284건 중 100건 골든 라벨링 (수동, 의사 1명 0.5~1일)
- LLM 프롬프트 설계 + 추출 실행
- 정확도·재현율·F1 측정
- 카테고리별 정확도 분포 분석

성공 기준: F1 ≥ 85% (도메인 한정이라 가능성 있음)

### 8.2 Phase 1 — 첫 가치 제품 1개 (3~4개월)

매핑 정확도 검증 후 다운스트림 제품 1개를 먼저 출시.

**후보**:

| 후보 | 가치 | 난이도 | 추천 우선순위 |
|---|---|---|---|
| 차팅 누락 알림 | 의사 즉시 가치 | 중 | **★ 첫 후보** |
| 청구 적정성 검증 | 운영진 가치 | 중 | ★★ |
| 마케팅 환자 타게팅 | 운영진 즉시 가치 | 낮 | ★★★ (가장 빠름) |
| 유사 사례 검색 | 의사 데모 임팩트 | 중상 | 두 번째 |

→ 마케팅 타게팅이 가장 빠르고 운영진 입장에서 ROI 명확. **출시 첫 제품으로 권장**.

### 8.3 Phase 2 — 가치 제품 추가 (6~12개월)

Phase 1 제품으로 시장 검증 완료 후 다른 제품들 추가:
- 청구 적정성 검증
- 차팅 누락 알림
- 유사 사례 검색
- 의사별 통계 대시보드

각 제품은 **같은 매핑 데이터 위에 올라간다** → 한계 비용이 낮음.

### 8.4 Phase 3 — 검증 UI + 추가 기능 (12~18개월)

매핑 신뢰도 낮은 항목에 대한 검증 UI 도입.
이때부터 옛 "클릭 차트"의 일부 기능이 자연스럽게 들어옴.

### 8.5 Phase 4 — 음성 입력 + Ambient AI (18개월+)

자유 텍스트 입력을 음성으로 확장. Nuance DAX 류 ambient AI 영역.
이미 추출 시스템과 다운스트림 제품들이 안정화된 상태에서 진입.

### 8.6 의사결정 분기점

| 시점 | 의사결정 |
|---|---|
| Phase 0 종료 | 매핑 정확도 < 80%면 전체 전략 재검토. 80~90%면 Phase 1 진행. 90%+면 가속. |
| Phase 1 출시 후 3개월 | 첫 제품 도입율 > 30%면 Phase 2. 미만이면 가치 명제 재검토. |
| Phase 2 종료 | 멀티 제품 LTV > CAC 검증. 통과 못 하면 Pivot. |

## 9부 · 사업성 재평가

### 9.1 단위 경제 (다시)

| 항목 | 추정 |
|---|---|
| 의사당 연 매출 | 100~200만원 (다운스트림 제품 묶음) |
| 의사당 연 변동비 | 30~80만원 (LLM + 인프라) |
| **단위 마진** | **+20~+170만원/년** |
| 개발 자본 (총합) | 5~10억원 |
| Break-even (의사 수) | 300~600명 |

이전 평가들과 비교:
- 클릭 차트 단독: +60~+90만원/년
- AI 추천 단독 (검색형): +170~+720만원/년
- **추출-우선 멀티 프로덕트**: +20~+170만원/년 × 더 많은 제품

→ 제품 개당 마진은 낮지만 **묶음 LTV가 더 큼**. 또한 **추출 인프라가 한 번 만들어지면 이후 제품 추가의 한계 비용이 낮음**.

### 9.2 시장 포지셔닝

기존 EMR 벤더와 차별화:
- 기존 EMR: 자유 텍스트 + 진단/처방 코드 입력 (구조화 거의 없음)
- 추가 SaaS: 음성 인식 차팅, ICD 검색 도구 (단편적)
- **CCF (재정의)**: 자유 텍스트 위에 자동 추출 + 멀티 다운스트림 제품 (구조화 데이터 활용 통합)

→ "EMR 보강 SaaS" 카테고리에서 차별화. EMR을 대체하지 않고 그 위에 얹힘.

### 9.3 경쟁 위협

- **기존 EMR 벤더**: 같은 기능 자체 개발 가능. 대응: 정형외과 깊이 + 빠른 출시
- **카카오·네이버 헬스케어**: 자본·LLM 우위. 대응: 의사 친밀도 + 한국 청구 통합
- **글로벌 ambient AI**: 한국어 한계로 진입 늦음. 대응: 한국 특화 + 1~2년 선점

## 10부 · 솔직한 결론

### 10.1 우리가 도달한 곳

지난 며칠 동안 우리는 다음을 발견했다:

1. **클릭 차트는 입력 도구가 아니라 표준화 도구**였다 (그동안 잘못 가정)
2. **의사들은 자유 텍스트를 영원히 이긴다** (30년 EMR 역사가 증명)
3. **LLM이 처음으로 자유 텍스트 → 표준어 매핑을 운영 가능 수준에 올려놓았다**
4. **표준화된 구조화 데이터는 의사·운영진·기관·산업 단위에서 수십 가지 가치 제품을 가능하게 한다**
5. **CCF의 진짜 자산은 클릭 UI가 아니라 표준 어휘 온톨로지**다

### 10.2 새 전략의 한 줄

> **"의사 입력은 자유 텍스트로 두고, AI가 백그라운드에서 표준 어휘로 매핑하며, 그 위에 다운스트림 가치 제품들을 쌓는다."**

### 10.3 이게 의미하는 것

- 클릭 차트는 **메인 제품이 아니다.** Layer 3의 선택적 검증 UI일 뿐.
- 가치는 **다운스트림 제품들의 묶음**에서 나온다.
- 차별화는 **표준 어휘 + 매핑 정확도 + 제품 다양성**의 조합.
- 진짜 자산은 **누적되는 표준화된 의료 데이터**이고, 이게 시간 갈수록 강해지는 해자.

### 10.4 무엇을 즉시 해야 하는가

1. **추출 PoC 진행** — 100건 골든 라벨링 + LLM 매핑 정확도 측정 (1~2주)
2. **첫 다운스트림 제품 결정** — 마케팅 환자 타게팅 또는 차팅 누락 알림 후보
3. **데이터 거버넌스 초안** — PHI 처리·동의 구조 법무 자문
4. **의사 인터뷰 5~10명** — 새 그림(자유 텍스트 + 백그라운드 추출)에 대한 반응 검증

### 10.5 무엇을 하지 말아야 하는가

- **클릭 차트 풀 UI 개발에 더 자원 투입 X** (사용성 함정)
- **자유 텍스트 그대로 임베딩에만 의존 X** (도메인 정확도 부족, 사용자 지적 정확)
- **LLM에 진단·처방 결정을 위임 X** (책임·환각 위험)

## 부록 — 의료 정보학 참고

이 전략은 의료 정보학에서 새로 발명된 게 아니라, **30년 묵은 문제에 LLM이라는 새 변수를 더한 것**이다. 관련 개념:

- **Computer-Assisted Coding (CAC)**: 자유 텍스트 → ICD/처방 코드 자동 추출. 미국에서 2010년대부터 시도. 정확도 한계로 부분 채택.
- **Natural Language Processing (NLP) in EMR**: 자유 텍스트 분석 시도 30년. 부분 성공.
- **Ambient Clinical Intelligence**: Nuance DAX, Abridge 등. 음성 → 차팅 자동 생성. LLM으로 본격화.
- **Health Information Exchange (HIE)**: 표준 어휘 기반 기관 간 의료정보 교환. SNOMED-CT, LOINC 등.

CCF의 추출-우선 전략은 이 흐름의 한국 정형외과 특화 적용이다. **새 발명이 아니라 적시 실행**이 차별화 포인트가 된다.
